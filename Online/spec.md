# RPS Extend - Online Match Specification (オンライン対戦仕様書)

このドキュメントは「ジャンケン・エクステンド」をオンラインで1対1対戦できるようにするための、ネットワーク接続方式、ステートマシン（状態管理）、および通信プロトコル（JSONペイロード）の仕様を定義したものです。

---

## 1. 接続方式（システムアーキテクチャ）

ゲームの特性（完全情報・同時ターン制）を満たすため、以下の2つのアプローチのいずれか、または両方をサポートする設計とします。

### 方式A: クライアント・サーバー方式（中央集権型）
中央のサーバーがゲームホストとなり、プレイヤーA・Bがそれぞれ接続する方式です。
*   **通信プロトコル**: 
    *   **CLIベースの場合**: サーバーに対して **SSH接続**（または生TCPソケット）。プレイヤーはターミナルから直接サーバーにログイン（または接続）してCUIでプレイします。
    *   **Webベースの場合**: **WebSocket** (または HTTP Long Polling)
*   **状態管理**: サーバーが完全に状態（両者の手札、点数、ターンの進行）を管理します。
*   **メリット**: 実装が容易。クライアント側には一切の情報を持たせないため、チート（相手の手札を覗き見る等）が物理的に不可能になります。同時出しの判定もサーバーが行います。

### 方式B: P2P方式（ピアツーピア）
プレイヤー同士が直接通信を行う方式です。
*   **通信プロトコル**: 
    *   **CLIベースの場合**: プレイヤーの一方がホスト（サーバー）となり、もう一方が**TCPソケット通信**で直接IP指定で接続します。
    *   **Webベースの場合**: **WebRTC (DataChannel)** を使用します。（マッチング用のシグナリングサーバーのみ別途必要）
*   **状態管理**: 互いに状態を持ち合い、同期します。
*   **メリット**: ゲーム専用のサーバーを維持・運用するコストがかかりません。
*   **チート対策（重要）**: 中央の審判がいないため、相手が手を出した後に自分の手を変える「後出しチート」を防ぐ仕組みが必要です。本仕様ではこれを **「コミット＆リビール方式」** によって解決します。

---

## 2. 秘匿情報の通信プロトコル（コミット＆リビール）

P2P方式を考慮し、**「相手の手札を知らない状態で、同時にお互いのカードを確定させる」** ための暗号論的コミットメント（Commitment Scheme）をプロトコルに組み込みます。

### フロー
1.  **Commit (確定)**: 
    *   プレイヤーはカードを裏向きで伏せる際、実際のカードデータではなく、カードIDにランダムな文字列（Salt）を結合したものの **ハッシュ値**（SHA-256など）を相手に送信します。
    *   例: `hash = SHA-256("Card:G_Red, Salt:xyz123")`
2.  **Wait**:
    *   両方のプレイヤーから Commit（ハッシュ値）が届くまで待機します。
3.  **Reveal (公開)**:
    *   両者のハッシュ値が出揃ったら、自分の「実際のカードデータ」と「Salt（ランダム文字列）」を相手に送信します。
4.  **Verify (検証)**:
    *   受け取ったカードデータとSaltを使ってハッシュ値を再計算し、最初に受け取ったCommitのハッシュ値と一致するかを検証します。一致しなければ「チート行為（後出し）」として切断・反則負けとします。

※ サーバー方式（方式A）の場合は、サーバーが両者の入力を待ってから同時に結果を通知するだけで良いため、このハッシュ化プロセスは省略可能です。

## 3. ターンの進行プロトコル（ステートマシン）

ゲームの進行状態は、各ターンごとに以下のフェイズ（State）を遷移します。両プレイヤーからの入力が揃った段階で次のフェイズへ移行します。

| フェイズ | 状態名 | 内容 | 通信ペイロードの例 |
| :--- | :--- | :--- | :--- |
| 0 | `MATCHING` | 対戦相手の接続待ち。接続完了後、手札の初期化を行う。 | `{"action": "ready"}` |
| 1 | `DUMMY_SET` | フェイズ1：ダミーカードの伏せ。 | `{"action": "commit_dummy", "hash": "..."}` |
| 2 | `EFFECT_SET` | フェイズ2：効果カードの公開。 | `{"action": "play_effect", "card": "Win_Rock"}` |
| 3 | `ABILITY_SET` | フェイズ3：能力カードの公開と処理。 | `{"action": "play_ability", "card": "Peek"}`<br>`{"action": "peek_target", "card_id": 5}` |
| 4 | `COMBAT_SET` | フェイズ4：勝負カードの伏せ。 | `{"action": "commit_combat", "hash": "..."}` |
| 5 | `REVEAL_COMBAT` | フェイズ5：勝負カードの一斉公開と勝敗判定。 | `{"action": "reveal_combat", "card": "R_Green", "salt": "..."}` |
| 6 | `GUESS_RESOLVE` | フェイズ6：探偵（Guess）能力の判定。（※盾で防がれていない場合） | `{"action": "guess_result", "is_correct": true}` |
| 7 | `TURN_END` | フェイズ7：カードの回収と捨て札処理。次のターンへ。 | `{"action": "turn_end_ready"}` |

※ 6ターン終了後は `GAME_OVER` 状態へ移行し、同点の場合はサドンデス用の `SUDDEN_DEATH` ステートへ遷移します。

## 4. データ構造（JSON仕様例）

通信でやり取りされるメッセージはすべてJSON形式とします。

### ベース構造
```json
{
  "type": "game_action",
  "turn": 1,
  "phase": "COMBAT_SET",
  "player_id": "player_A",
  "payload": { ... }
}
```

### アクション別ペイロード例

**ダミー/勝負カードのコミット (Commit)**
```json
{
  "action": "commit",
  "target": "combat", 
  "hash": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
}
```

**カードの公開 (Reveal)**
```json
{
  "action": "reveal",
  "target": "combat",
  "card": "ROCK_RED",
  "salt": "random_salt_8a9b"
}
```

**能力カード：探偵（Guess）の宣言**
```json
{
  "action": "play_ability",
  "card": "GUESS_HAND_COLOR",
  "guess_target": "SCISSORS_BLUE"
}
```

## 5. エラーハンドリング・切断時の処理

*   **タイムアウト**: 各入力フェイズには制限時間（例: 60秒）を設けます。制限時間内に入力がない場合はランダム選択、またはタイムアウト負けとします。
*   **通信切断**: 通信が切断された場合、再接続のための猶予時間（例: 30秒）を設けます。猶予時間内に再接続できなかった場合はサレンダー（降伏）扱いとします。
*   **不正検知**: コミット＆リビールの検証に失敗した、または手札にないカードを使用しようとした場合は、そのメッセージを送信したプレイヤーを反則負けとします。
